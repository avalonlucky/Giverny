## 這是什麼

Giverny 不是又一個泛用專案管理工具。它更像一個「協同任務 + 工時證據 + 交付驗收 + 月度結算」的輕量營運後台，適合需要把設計、內容、營運、外包或內部協作任務從需求到結算完整留痕的團隊。

它關心的不是「任務有沒有被拖到某個看板列」，而是每一項工作最終能不能回答清楚：

- 這件事是誰提的、誰對接、什麼時候交付？
- 過程裡做了哪些推進，上傳過哪些檔案？
- 實際投入了多少工時，哪些工時可以結算？
- 最終由誰驗收，驗收依據是什麼？
- 月底給客戶或公司對帳時，能不能一鍵拿出任務、工時、檔案和金額？

## 核心能力

| 能力 | 解決的問題 |
| --- | --- |
| 任務流轉 | 從計畫中、進行中、待驗收到已驗收，完整記錄任務狀態 |
| 過程協同 | 在任務右側記錄進展、附件、時間段和動態時間軸 |
| 工時管理 | 所有統計、收入和結算只基於實際工時，不用預計時間湊數 |
| 交付驗收 | 透過終審面板核對基礎資訊、進度、分段工時、附件和備註 |
| 月度結算 | 按結算月份彙總工時、收入、驗收情況和年度趨勢 |
| 客戶對帳 | 生成唯讀分享連結，讓甲方只看月報、任務明細和交付檔案 |
| 檔案歸檔 | 把過程檔案、驗收附件和最終稿沉澱到任務生命週期裡 |

## 適合這些場景

- 公司內部設計、內容、營運、市場任務需要跨人協同和月度複盤。
- 兼職設計師或外包團隊需要記錄實際工時、交付檔案和結算依據。
- 負責人需要按月核對「做了什麼、花了多久、交付了什麼、該結算多少」。
- 甲方或協作方需要一個唯讀頁面查看月報和交付內容，而不是翻聊天記錄。

## 專案定位

Giverny 是一個用於管理設計兼職工作的營運後台。它把任務需求、設計過程、實際工時、過程檔案、驗收附件、月度結算和甲方唯讀對帳連結放在同一個系統裡，減少“聊天記錄裡找需求、表格里算工時、網盤裡找檔案、月底再手工做帳”的反覆切換。

它目前服務於正式站 [mayeai.com](https://mayeai.com)。正式站只放真實營運資料，使用 Cloudflare D1 儲存業務資料，使用 Cloudflare R2 儲存上傳檔案。預發布測試站已經下線，後續功能完成本機驗證後直接更新正式站。

Giverny 不是泛用專案管理工具。它更偏向一個“設計服務結算工作台”：圍繞設計任務的生命週期，把過程、工時、檔案、驗收和月報串起來。

## 為什麼需要 Giverny

設計兼職工作常見的問題不是“沒有任務管理工具”，而是資訊被拆散在太多地方：

- 需求在微信、飛書、郵件和口頭溝通裡。
- 工時在臨時 Excel 或聊天備註裡。
- 過程檔案和最終稿混在網盤或本機資料夾里。
- 甲方月底只關心“做了什麼、花了多久、交付了什麼、為什麼要結算這筆錢”。
- 設計師自己又需要留痕：什麼時候接的需求，什麼時候改過，哪個檔案是最終驗收依據。

Giverny 的目標是把這些分散資訊收斂成一條可追溯鏈路：

```text
任務需求 → 過程進展 → 時間記錄 → 檔案歸檔 → 交付驗收 → 月度結算 → 甲方唯讀對帳
```

## 適合誰

| 角色 | 主要訴求 | Giverny 提供什麼 |
| --- | --- | --- |
| 兼職設計師 | 清楚記錄每項需求、過程修改、實際工時和驗收檔案 | 任務詳情、進展時間軸、分段工時、驗收附件和月度統計 |
| 設計服務負責人 | 按月核對工作量、收入、檔案和結算狀態 | 工作台、收入統計、檔案库、月報和 PDF |
| 甲方 / 協作方 | 只看最終月報、任務明細和交付檔案，不進入後台 | `/share/:token` 唯讀連結 |
| 接手開發者 / AI 编码助手 | 快速理解程式碼結構、業務規則和發布紀律 | `AGENTS.md`、`docs/`、`handoff/`、Releases |

## 典型使用場景

### 場景 1：本月正常任務

1. 在目前月份新建任務，填寫任務名稱、設計類型、需求、預計開始和預計交付。
2. 接受任務後在右側“進展”面板記錄過程，例如“已確認尺寸，正在出草圖”。
3. 需要記錄實際投入时，新增時間段，例如 `09:00 - 10:30 初稿`。
4. 上傳過程附件或最終輸出檔案。
5. 到交付时展開“交付驗收”，核對基礎資訊、进度、工時、附件和備註。
6. 確認驗收後，任務计入本月工時、收入和月報。

### 場景 2：補錄歷史任務

1. 新建任務时開啟“補錄”。
2. 任務日期仍填寫真實發生時間，例如 5 月 20 日。
3. 結算月份選擇需要计入的月份，例如 2026 年 6 月。
4. 甲方看到“補錄”標記後，就能理解這條任務為什麼出現在本月結算中。
5. 補錄是公開解釋資訊，不能做成管理員專屬棕色資訊。

### 場景 3：任務已做完但還沒驗收

1. 狀態保持“待驗收”。
2. 在任務右側補充實際工時、交付附件和驗收備註。
3. 點擊“去驗收”開啟終審彈窗。
4. 確認後狀態变为“已驗收”，进度自動到 100%，工時鎖定并進入結算。

### 場景 4：甲方月底對帳

1. 管理員在“月報 / 結算”里鎖定月度資料。
2. 生成唯讀分享連結。
3. 甲方透過連結檢視本月任務、工時、交付檔案和結算金額。
4. 甲方頁面不顯示後台編輯入口，也不能刪除、修改或驗收任務。

## 核心工作流

![Giverny 任務生命週期](./docs/assets/readme/workflow.svg)

1. **新建任務**：記錄任務名稱、設計類型、需求說明、預計開始、預計交付、對接人和結算月份。
2. **過程推进**：在任務右側“進展”面板記錄進展、上傳過程附件、追加時間段、維護狀態和整體进度。
3. **工時沉淀**：所有統計以“實際工時”为准；預計開始時間和預計交付時間只用於排期参考，不參與資料分析、工時计算或結算。
4. **交付驗收**：展開交付驗收面板，核對基礎資訊、进度、分段工時、驗收附件和備註；確認後狀態变为“已驗收”，进度鎖定为 100%，工時计入結算。
5. **月度結算**：按結算月份彙總工時、收入、驗收情况和年度趨勢，生成唯讀甲方連結和 PDF。

## 模組詳解

### 工作台

工作台是每个月的營運总览，核心問題是：“這個月做了多少、能結算多少、哪些還沒驗收？”

它展示：

- 本月总工時：本月任務實際投入工時。
- 計費工時：已驗收或符合結算口徑的工時。
- 預計收入：按设置中的时薪计算。
- 驗收情况：已驗收 / 总任務数，以及待驗收数量。
- 逾期提醒：把預計交付已過期的任務集中提示。
- 任務明細：按狀態篩選本月任務。
- 本月洞察：基於實際工時生成設計類型分布和周趨勢。
- 年度統計：按月彙總全年工時与收入。

關鍵口徑：工作台所有統計都基於 `settlement_month` 和實際工時，不使用預計開始 / 預計交付參與結算。

### 任務導航

任務導航負責日常維護任務。左側是列表或日曆，右側是選中任務的詳情。

列表侧：

- 支援“全部 / 計畫中 / 进行中 / 掛起 / 待驗收 / 已驗收 / 終止”篩選。
- 展示任務名、需求摘要、對接人、交付時間、工時和狀態。
- 補錄任務會顯示公開“補錄”標籤。
- 管理員可透過右鍵選單檢視詳情、變更狀態、複製任務名、複製甲方分享連結、作廢或恢復任務。

右側資訊页：

- 展示任務名稱、設計類型、預計開始、預計交付、結算月份、需求人、對接人、驗收人和任務需求。
- 預計開始 / 預計交付只用於排期，不參與統計和結算。
- 只给管理員看的內部資訊使用棕色 `admin-only-data`。

右側進展页：

- 過程記錄：記錄“现在做到哪一步”。
- 進展附件：上傳過程檔案或溝通附件。
- 時間記錄：新增分段工時，例如初稿、终稿、修改。
- 整體進展：以 10% 檔位調整，先進入未儲存草稿，確認後才寫入時間軸。
- 動態時間軸：記錄任務狀態、进度、附件、排期等變更。
- 交付驗收：專案收尾时展開，進入終審確認。

### 交付驗收

驗收是任務從“過程”進入“結算”的邊界。終審彈窗用於讓使用者逐項核對：

- 基礎資訊：任務名稱、設計類型、對接人、預計開始、預計交付、任務需求。
- 进度：顯示目前完成百分比；確認驗收後自動设置为 100%。
- 分段工時：可以核對或修改每段時間，最終得到實際工時合计。
- 驗收附件：上傳驗收證明檔案或最終交付檔案。
- 驗收備註：記錄補充說明。

確認驗收後：

- 狀態变为“已驗收”。
- 實際工時鎖定進入結算。
- 进度变为 100%。
- 本專案结束，并计入工作台、收入、年度統計和月報。

### 檔案库

檔案库不是獨立網盤，而是任務生命週期的檔案歸檔檢視。檔案來源包含：

- 任務過程檔案。
- 進展附件。
- 驗收附件。
- 最終稿。

檔案會按任務和專案歸檔，支援預覽、開啟源檔案、下載、重新命名、新增標籤和刪除誤傳檔案。

刪除規則：

- 只用於清理誤傳、重複上傳或無價值臨時檔案。
- 刪除必須使用站内二次確認。
- 刪除会清理 D1 附件記錄、R2 源檔案和預覽圖。
- 刪除動作寫入稽核日志。

### 收入与月報

收入頁用於檢視收入趨勢和稅後估算。月報用於對帳和分享。

月報會讀取鎖定月份的資料，生成唯讀連結。甲方透過連結可以看到：

- 月度任務彙總。
- 任務明細。
- 工時和結算金額。
- 交付檔案。
- PDF 匯出内容。

甲方連結是唯讀的，不提供任何管理操作。

### 设置

设置页管理平台參數：

- 口令管理：生成和管理存取口令。
- 設計類型：維護設計類型大类和子类。
- 时薪与计税方式：影響收入估算。
- PDF 抬頭与服務公司名稱：影響結算回单展示。
- 帳號安全：修改管理員密碼。
- 系統資訊：檢視版本、Cloudflare 绑定和备份入口。

- 甲方唯讀頁面不顯示後台按鈕、刪除入口、內部時間和管理狀態。

## 資料口徑與結算規則

### 月份歸屬

任務歸屬月份只看 `settlement_month`。這意味著：

- 普通任務預設歸屬頂部目前選擇的月份。
- 補錄任務可以把真實發生日期和結算月份分开。
- 工作台、任務列表、收入統計、月報都按 `settlement_month` 統計。

### 實際工時

實際工時來自：

- 分段時間記錄彙總。
- 驗收时確認的實際工時。
- 已驗收任務鎖定後的工時。

實際工時會參與：

- 本月洞察。
- 收入估算。
- 月報。
- 年度統計。
- 結算 PDF。

### 預計開始 / 預計交付

預計開始時間和預計交付時間只用於排期与提醒：

- 可以用於判断临期 / 逾期。
- 可以幫助任務排序和日曆檢視。
- 不參與工時統計。
- 不參與收入计算。
- 不參與結算月份兜底。

## 檔案生命週期

```text
上傳過程檔案 → 自動進入檔案库 → 標記用途 / 標籤 → 驗收附件或最終稿 → 月報唯讀分享 → 长期保留或人工清理
```

目前沒有自動定期清理机制。原因是歷史附件可能被月報、甲方分享页、復盤或返修繼續引用。正式營運階段建議優先“歸檔”而不是直接刪除。

推荐清理策略：

1. 月報鎖定後再清理。
2. 最終稿、驗收附件、合約或結算相關檔案優先保留。
3. 重複上傳、誤傳檔案可以刪除。
4. 大型過程源檔案後續可增加“待歸檔 / 待刪除 / 最終清理”三段式机制。

## 部署與發布

部署前请閱讀 [docs/DEPLOYMENT.md](./docs/DEPLOYMENT.md)。目前專案不再維護 staging 站，完成本機驗證後直接部署正式站。

```bash
env -u ALL_PROXY -u HTTPS_PROXY -u HTTP_PROXY -u all_proxy -u https_proxy -u http_proxy npm run deploy:worker
```

正式資源：

- Worker：`designer-worklog`
- D1：`designer-worklog-db`
- R2：`designer-worklog-uploads`
- 網域：`mayeai.com` / `www.mayeai.com`

正式更新必須完成：

1. 更新版本號：`src/config/appConfig.ts`、`package.json`、`package-lock.json`。
2. 更新 `CHANGELOG.md`，必要時更新 `使用手册.md` 和相關 docs。
3. 執行 `npm run lint` 和 `npm run build`。
4. 部署正式站并驗證線上資源版本。
5. `git commit`、`git push`。
6. 建立并推送 tag，例如 `v0.10.69`。
7. 建立 GitHub Release，Release notes 按“大更新 → 小更新”排序。
8. UI / 互動明显变化时，Release 需要上傳截圖或圖示資產。

## FAQ

### 為什麼不直接用 Notion / 飛書表格 / Excel？

這些工具能記錄資訊，但很难把“任務進展、實際工時、附件、驗收、結算、甲方唯讀連結”穩定串成一條可稽核鏈路。Giverny 的價值在于圍繞設計服務結算做专门约束。

### 為什麼任務不能直接刪除？

任務會影響工時、收入、月報和歷史對帳。直接刪除会破壞歷史資料。異常任務應使用“掛起”“終止”或“作廢”來保留解釋。

### 為什麼預計交付不參與結算？

預計時間是計畫，不是投入。結算必須基於實際工時和驗收狀態。預計開始 / 預計交付只用於提醒、排序和排期参考。

### 為什麼補錄要让甲方可见？

補錄解釋了“這條任務為什麼出現在本月結算裡”。如果隱藏，甲方可能會認為本月憑空多了一條需求。

### 為什麼管理員專屬資訊是棕色？

棕色是後台內部資訊標記。管理員看到棕色就知道這類資訊不會出現在普通成員、甲方預覽或公開唯讀連結裡。
