# AI Learning

Giverny 的“持续学习”不是让模型在生产环境自行训练或修改规则，而是把可验证的人工反馈沉淀为结构化数据，再用于后续检索、风格归纳和数值校准。

## 学习闭环

```text
原始输入
  -> AI 建议
    -> 用户直接采用 / 修改后采用 / 不采用
      -> 最终任务、进展、反馈、验收或文件名
        -> ai_learning_events
          -> 增量风格摘要 / 历史相似样本 / 工时偏差校准
            -> 下一次 AI 建议
```

`ai_learning_events` 保存：

- `context`：任务需求、任务名称、任务类型、工时、进展、修改意见、验收备注或附件命名。
- `source_input`：生成建议前的用户原始输入。
- `ai_output`：当次 AI 建议。
- `user_final`：最终保存到业务数据中的文本或选择。
- `action`：`adopted`、`edited` 或 `rejected`。
- 任务、设计类型和少量结构化 metadata，用于按场景归纳，不用于公开展示。

## 当前覆盖

| 场景 | 学习方式 |
|---|---|
| 任务需求 | 对比原始需求、AI 三段式任务单和最终提交文本；未点击采用但最终保存的人工写法也会进入偏好样本 |
| 任务名称 | 对比建议名称与最终名称，按设计类型归纳命名习惯 |
| 任务分类 | 保存 AI 推荐分类和最终人工分类，后续作为分类 few-shot 示例 |
| 预估工时 | 保存需求质量、复杂度画像、语义相似样本、可解释拆分、采用值、真实工时和报价结果；验收时生成需求变更时间线和偏差复盘，再按近期同类样本、个人采用、需求人画像、样本质量和个人效率受控校准 |
| 进展记录 | 学习进展内容的分点、措辞、事实边界和下一步写法 |
| 修改意见 | 使用独立提示词与风格画像，保留版本、来源、修改对象和关键约束 |
| 验收备注 | 学习面向甲方的项目总结写法，并继续读取任务需求、全部历史进展和验收附件分析 |
| 附件命名 | 对比建议文件名与最终文件名，按设计类型归纳命名习惯 |

## 安全边界

- 学习事件只记录建议与最终业务结果，不允许模型直接改提示词、数据库结构、结算数据或权限策略。
- 风格摘要至少积累 3 条可比较样本后才生成，后续每次最多处理 30 条新增样本，避免无限增长 prompt。
- 新样本与旧偏好冲突时，由受控摘要任务按最新样本更新；原始事件保留，便于审计。
- 工时只使用已验收任务的真实工时做结果校准，预计日期跨度不作为实际工作量。
- 需求人独立系数至少需要 3 条完整样本，且限制在受控范围内；样本不足时只展示观察，不调整建议值。
- 同一设计类型至少需要 3 次可比较采用结果才启用个人采用校准，系数限制在 `0.85–1.2`；同一次建议的重复反馈会去重。
- 历史工时样本按时间温和衰减，近期任务优先，但老任务不会被直接删除。
- AI 建议始终是辅助输入，任务创建、进展保存、验收和写入 Agent 仍由用户明确操作或确认。

## 工时学习可观测性

「洞察 → AI 工时学习 → 准确率与校准」把工时学习过程转成可核对指标：

- 只统计已验收并有真实工时的预测，避免用未完成任务制造命中率。
- 分开统计常规值、稳妥值和手工修改后的误差，验证用户采用行为是否真的改善结果。
- 按设计类型和需求方 / 客户形成校准分组；至少 3 条样本后才允许独立系数影响后续建议。
- 每条验收结果生成逐任务复盘，偏差因素只读取分段计时、等待、改稿、需求画像和反馈标签等业务事实。
- 语义相似任务仍由 Vectorize 召回，但最终排序会交叉验证类型、复杂度、交付规模和近期性，向量相似度不是唯一依据。
- 最近 12 个月会按月比较命中率、中位误差、低估率和高估率，避免只用单月快照判断是否进步。
- 用户可把错误参考样本标记为不相似；系统只在相近类型和复杂度区间内降低权重，不会删除历史事实。
- 显著需求扩张会单独记录，并从预测偏差和需求人系数中排除，避免把范围变化学成效率问题。
- 管理员可以人工校正偏差原因；每次建议同时保存算法、提示词和模型路由版本，后续按真实验收结果比较版本。
- 建议前先评估需求质量；信息不足时只给区间或要求补充，不允许一键采用伪精确数字。
- 异常工时、过短需求和范围变化进入样本质量中心；人工排除只降低后续权重，不删除业务事实。
- 报价、报价接受状态和实际结算金额单独形成定价反馈，不与工时准确率混为一个指标。
- 新版本发布前使用按时间顺序的无未来数据回放；候选中位误差超出线上基线容差时应阻止发布。

## 当前学习版本

AI 工时算法 `3.1.0` 在 3.0 的安全边界上新增六条可验证链路：

- 以 20 条“健康验收 + 报价结果”作为首轮完整生命周期观察门槛，样本不足时明确标记采集期或校准期。
- 按设计类型与从零 / 复用基础诊断中位误差、低估、高估和集中偏差因素。
- 根据当前需求缺口提供结构化补全选项，补入需求后必须重新分析。
- 同类型最近 3 条与此前 3 条真实工时偏移 20% 以上时生成漂移提醒。
- 将工时、报价接受状态与最终结算交叉分析，形成可解释定价建议。
- 与上一次同类型、同需求方建议比较，审计复杂度、需求质量、样本和校准变化。

这些能力仍是受控统计和辅助决策，不会自行修改任务、报价、结算或模型规则。
