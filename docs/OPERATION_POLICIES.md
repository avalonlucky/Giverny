## 1. 每次更新后的文档要求

每次功能更新后，本地必须同步维护以下文档：

1. `CHANGELOG.md`
   - 记录具体时间，格式建议：`YYYY-MM-DD HH:mm`。
   - 按“新增 / 修复 / 优化 / 运营规范 / 部署”分类描述。
   - 必须写清楚是否已经同步线上、是否影响数据结构或权限。

2. `使用手册.md`
   - 面向实际使用者，不写太多工程细节。
   - 新增功能、删除功能、入口变化、权限变化、结算规则变化，都需要同步更新。
   - 如果某个功能还只是本地验证，不要写成正式站已可用。
   - 核心功能不能只写“新增了什么”，还要解释入口、适用场景、操作步骤、结果怎么看、数据来源、计算 / 判断原理和安全边界。
   - AI、结算、权限、数据学习等高影响功能，应给出完整生命周期和可验证示例；普通 UI 微调和 bug 修复只记录用户可感知变化，不需要重复写技术原理。
   - 独立机制较复杂时，在 `docs/` 下新增专题功能文档，并从使用手册链接过去；专题文档负责精确口径，使用手册负责向用户讲清楚。

3. `docs/DEPLOYMENT.md`
   - 只在部署流程、Cloudflare 资源、上线规则变化时更新。

4. `docs/OPERATION_POLICIES.md`
   - 记录排序逻辑、数据保留、文件清理、误操作防护等长期规则。

## 2. 任务排序逻辑

当前工作台和任务模块使用同一套月份归属与排序来源：

```sql
SELECT * FROM tasks
WHERE settlement_month = :selected_month
  AND deleted_at IS NULL
ORDER BY start_date DESC, created_at DESC
```

也就是说：

- 月份筛选口径：补录任务以结算归属月份 `settlement_month` 为准；普通任务的收入、工时、月报和回单按分段计时实际发生月份拆分。预计开始时间和预计交付时间只作为排期参考，不能作为金额归属兜底。
- 第二优先级：预计开始时间 / 开始日期，越新的日期时间越靠上。
- 第三优先级：同一天内按创建时间倒序，后创建的任务更靠上。
- 不是按预计交付日期排序。
- 不是按任务状态排序。
- 不是手动拖拽排序。

### 对用户的表现

- 新建任务如果预计开始时间是今天，它会排在今天这一组靠前位置。
- 普通任务的结算月份默认等于顶部当前选择的结算月份，不跟随预计开始时间变化。
- 如果新建任务时把预计开始时间填成上个月，且没有打开「补录」，它仍会归到当前选择的结算月份；需要改归属月份时，应显式修改「结算月份」。
- 如果任务实际 5 月完成，但工资需要放入 6 月结算，应在新建任务弹窗右上角打开「补录」开关，把结算月份选为 6 月。此时任务会出现在 6 月工作台、6 月任务列表、6 月收入统计、6 月月报和甲方 6 月分享页中；任务日期仍显示真实的 5 月日期，并标记为补录。
- 已完成、只需要补结算的历史任务，新建时填写补录说明即可；创建后进入任务详情点击「确认验收」，再补实际工时、验收备注和验收文件。新建入口不展示阶段选择，也不直接操作验收。
- 补录任务只归属明确指定的 `settlement_month`，原工作分段月份和补录操作日期不能让它重复出现在其他月份。若存在验收进展分段，实际完成日期与列表最新动态必须优先采用该分段结束时间；预计交付日期只用于排期风险，不参与真实动态日期计算。
- 工作台、任务列表、收入统计、月报统计都会使用统一财务口径：补录任务完整计入 `settlement_month`，普通跨月任务按每段进展的发生月份拆分；任务保存的实际工时是整单总量锚点，分段分钟与已保存工时存在舍入差异时也不得替换该锚点，拆分后的各月份金额相加必须等于原任务总额。只有已保存工时为 `0` 时，才允许从有效分段补算。

### 后续可优化方向

如果正式使用时觉得“开始日期倒序”不够顺手，可以增加排序选择：

- 最近创建优先
- 预计交付临近优先
- 状态优先：进行中 > 待验收 > 计划中 > 已验收 > 挂起 / 终止
- 手动置顶

正式运营阶段建议先保持当前统一规则，不要同时引入多套排序，避免月报核对时产生理解偏差。

## 3. 文件库清理机制

### 当前状态

当前没有自动定期清理机制。

文件库中的文件来自任务生命周期：

- 任务过程文件
- 进展附件
- 验收附件
- 最终定稿

文件删除用于处理误传文件：

- 前端文件库右侧信息栏提供删除图标。
- 点击删除后必须通过站内二次确认弹窗。
- 确认后会删除 D1 `attachments` 记录、R2 源文件和 R2 预览图。
- 删除动作写入审计日志。

### 为什么不建议自动按月删除

- 月报和甲方分享页可能仍需要访问历史附件。
- 已验收任务未来仍可能返修、补交或复盘。
- 自动删除很难判断“过程文件”和“仍有价值文件”的边界。
- 一旦删除 R2 源文件，恢复成本很高。

### 推荐清理流程

每月结算后执行一次人工清理复核：

1. 确认本月月报已锁定。
2. 确认甲方已验收或不再需要查看过程文件。
3. 导出备份 JSON。
4. 检查文件库中该月文件，区分：
   - 必须保留：最终稿、验收文件、合同 / 结算相关文件、甲方已查看文件。
   - 可归档：大型 PSD / AI / 视频过程源文件。
   - 低价值临时文件：重复上传、错误上传、无价值过程文件。当前阶段也先保留，后续进入“待删除”流程。
5. 仅删除明确误传、重复上传或无价值的临时文件；已验收、已发给甲方或可能用于回溯的文件优先保留。

### 更保险的后续产品机制

建议后续增加三段式清理，而不是直接删除：

1. 标记为“待归档”
   - 不从 R2 删除，只在文件库中降低优先级。
2. 标记为“待删除”
   - 进入待删除列表，保留 30 天。
3. 最终清理
   - 超过保留期后，管理员手动确认批量删除。

同时保留以下保护：

- 当月文件默认禁止删除，除非明确勾选“我确认这是误传文件”。
- 已锁定月报引用过的文件默认禁止删除，只允许归档。
- 最终稿、验收文件默认禁止删除，只允许管理员改状态。
- 删除动作写入审计日志。

## 4. 正式站发布规则

- 正式站：`https://mayeai.com`
  - 只录入真实任务、真实文件、真实结算数据。
- 预发布测试站已下线，测试 D1/R2 资源也已删除。

每次功能变更流程：

1. 本地开发。
2. `npm run lint`
3. `npm run build`
4. 高风险改动先在本地或临时隔离环境验证清楚。
5. 部署正式站。
6. 按 `docs/VERSIONING.md` 判断版本号递增类型，并同步更新 `src/config/appConfig.ts`、`package.json` 和 `package-lock.json`。
7. 更新 `CHANGELOG.md` 和 `使用手册.md`。

## 5. 试运营期间的数据安全原则

- 任务不允许删除，避免破坏工时和收入统计。
- 文件允许管理员二次确认后删除误传文件；常规历史文件优先保留，后续再补归档机制。
- 月报锁定后应视作财务快照，不随意改历史数据。
- 所有大改版先做好本地或临时隔离验证，再部署正式站。
- 正式站数据出现异常时，先导出备份，再排查问题。
