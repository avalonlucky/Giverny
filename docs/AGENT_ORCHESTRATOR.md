# Agent Orchestrator

## 定位

Giverny Agent 不把大模型直接等同于产品。大模型负责理解、规划和语言组织；Agent 编排层负责权限、工具、业务规则、证据、确认、重试和结果验真。

统一链路：

```text
用户请求
  -> AgentTurn（租户、身份、会话、本轮目标）
  -> 理解意图
  -> 生成计划
  -> 权限与风险授权
  -> 调用 D1 / R2 / 产品注册表 / 知识库工具
  -> 模型分析
  -> 确定性验真
  -> 证据不足或工具失败则动态重规划（最多三轮）
  -> 仍不满足则停止或追问，不编造业务结论
  -> 写入动作进入预览与人工确认
  -> 最终回答
```

## 核心契约

`src/agentOrchestrator.ts` 定义统一 `AgentTurn`：

- `principal`：工作区、登录主体、角色、本轮运行 ID。
- `intent`：财务、任务、附件、产品帮助、知识、写入或普通问答。
- `plan`：模型提出的工具调用计划及理由。
- `evidence`：工具返回的结构化证据，标明来源和是否可确定性校验。
- `verification`：业务结论是否有足够证据、缺少哪些工具、是否需要继续执行。
- `phase`：理解、计划、授权、执行、分析、验真、完成、待补充或失败。
- `status`：每个工具调用记录待执行、成功、跳过、拒绝或失败，以及尝试次数和耗时。

## 动态重规划

编排层不会把模型首轮计划当成不可修改的指令。每次工具执行后先进入验真：

1. 工具临时失败时，保持用户所选主模型和原业务目标，最多重试三次。
2. 财务、任务阻塞等必需证据缺失时，根据验真器返回的 `requiredTools` 生成补查计划。
3. 规划模型多选但与当前目标无关、参数不成立的工具会标记为 `skipped`，不会被当成执行故障。
4. 三轮后仍缺少必要证据时，本轮进入 `needs_input` 或 `failed`，禁止把模型猜测包装成业务事实。

## 模型与 Runtime

自动模式、工作助手手选云模型和本机 CLI 必须服从同一套 AgentTurn、工具注册表、权限与验真规则。模型只是可替换的推理适配器：

- 用户手选模型在正常情况下始终是本轮主模型。
- 主模型先按同模型策略完成重试和结构修复。
- 只有超时、额度、鉴权、供应商故障或多次无效结构等极端失败，才启用备用模型。
- 备用模型切换必须在运行记录中标注实际模型与原因，不能静默掩盖主模型故障。
- 本机 CLI 不持有 D1/R2 凭证，只能使用当前网页会话签发的短期、只读、租户受限工具。

## 权限与租户隔离

`src/agentScope.ts` 为内部工具调用生成 HMAC 签名上下文：

- `workspaceId`
- `principalId`
- `role`
- `runId`

Worker 会同时校验内部密钥、签名和工具权限。仅修改工作区 Header 而不重新签名会被拒绝。任务、附件、财务、会话、计划和任务记忆都必须使用该工作区过滤。

角色规则：

- `admin` / `collaborator`：可读；写入仍必须先预览再确认。
- `viewer` / `mcp-read`：只读。
- `client`：只读允许范围内的任务和附件，不开放管理员财务工具。
- `guest`：仅产品说明与公开能力上下文。

## 验真规则

- 金额、工资、计费工时必须来自财务计算工具，不允许模型自行重算。
- 具体任务卡点、延期、等待原因必须读取任务详情和有效等待记录。
- 附件存在性、名称、类型和地址必须来自附件工具。
- 工具返回多个相似任务时必须让用户选择。
- 业务事实没有确定性证据时不得输出肯定结论。
- 写入必须经过预览、签名确认凭证、人工确认和幂等执行。

模型若漏掉必需工具，编排层会补查；模型回答与工具证据冲突时，工具事实优先，并记录一次验真纠正。

## 执行审计

每轮执行会写入 `agent_turn_runs`，并在「设置 → 模型 → AI 运行中心」展示。审计只保存：工作区、主体、Runtime、模型、意图、工具名与状态、证据来源、验真结果、尝试次数、耗时和备用模型原因；不保存用户问题、回答正文或模型隐藏思维链。自动化评测记录独立标记，不进入真实运营统计。

## 多租户开放门槛

正式开放注册前至少满足：

1. Agent 所有业务工具都有工作区过滤回归。
2. 会话、计划、记忆、后台任务和模型配置全部租户化。
3. 跨租户 ID、标题模糊搜索、附件路径和确认凭证均无法越权。
4. 所有写入工具均有人审确认、审计记录和幂等保护。
5. 主模型故障、工具超时、证据不足和备用模型切换均有可观测记录。
6. 关键业务问题有隔离评测和浏览器回归，不依赖人工碰运气发现。
