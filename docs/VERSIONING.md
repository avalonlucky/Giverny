# Giverny 版本号规范

当前版本：v0.11.26
发布时间：2026-06-20 02:53
阶段：试运营

## 版本格式

采用 `v主版本.次版本.修订版本`，例如 `v0.10.0`。

试运营期间主版本保持 `0`，表示产品仍在快速迭代。等系统稳定、正式作为长期运营平台使用时，再升级到 `v1.0.0`。

## 递增规则

### 主版本：`v1.0.0`、`v2.0.0`

用于里程碑或不兼容变化：

- 正式从试运营进入稳定运营。
- 数据结构或权限体系发生重大变化，需要迁移或重新确认使用方式。
- 月报、结算、税务、甲方分享等核心流程出现不兼容调整。

### 次版本：`v0.11.0`

用于可感知的新功能或核心流程升级：

- 新增独立模块，例如收入、文件库、部署体系、版本体系。
- 新增影响工作流的能力，例如补录结算月份、确认验收流程、甲方分享。
- 大范围 UI / 信息架构调整，例如导航重组、详情页重构、右键菜单体系化。

### 修订版本：`v0.10.1`

用于小优化、修复和文案调整：

- 修 bug、修样式错位、修按钮不可点。
- 局部交互优化，例如按钮图标化、弹窗样式优化。
- 网站内用户可见文案、提示语、默认值的小调整。
- 不改变数据库结构和核心业务流程的小改动。

### 不递增版本

以下内容不影响网站本体，不需要递增版本号，不需要 tag，不需要 GitHub Release：

- README、仓库首页、多语言 README、徽章、截图排版等 GitHub 展示资料。
- handoff、内部协作说明、AI 接手说明等只服务开发协作的文档。
- 不改变网站运行、不改变用户界面、不改变部署资源的纯说明文字整理。

## 每次更新的执行要求

1. 先判断本次更新是否影响网站本体。只有影响功能、UI、交互、数据口径、部署配置、数据库 / R2 / Worker 行为或用户可见文案时，才进入版本流程。
2. 如果只是 README / 仓库展示 / 内部协作文档，普通提交即可，不执行版本、tag、Release 流程。
3. 判断网站更新属于主版本、次版本还是修订版本。
4. 同步更新：
   - `src/config/appConfig.ts` 中的 `appVersion`、`appReleaseDate`、`appReleaseStage`
   - `package.json`
   - `package-lock.json`
   - `CHANGELOG.md`
   - 必要时更新 `使用手册.md`
5. `CHANGELOG.md` 的最新条目必须写明版本号和发布时间。
6. `CHANGELOG.md` 条目按影响大小排序：先写用户最容易感知的大更新 / 核心流程变化，再写小的样式、文案、规则和维护项。
7. 线上部署后，用设置页或左侧底部版本号确认是否显示为最新版本。
8. 每次网站正式更新完成后，必须同步完成 GitHub 发布闭环：
   - 提交代码：`git commit`
   - 推送代码：`git push`
   - 创建版本标签：`git tag vX.Y.Z` 并推送 tag
   - 创建 GitHub Release，并把本版本更新日志写入 Release notes
   - GitHub Release 的内容也按“大更新 → 小更新”排序，避免用户先看到零碎细节。
9. 如果本次更新包含明显 UI / 交互变化，Release 里应附上必要截图或动图，让设计师、甲方、协作者能直观看到改了哪里、怎么使用、有什么影响。
10. 如果是影响网站行为的配置或内部规则更新，可以不附截图，但 Release notes 仍要写清楚变更原因和影响范围。

## 显示位置规范

- 设置页显示版本号和发布时间。发布阶段只保留在版本规范文档中，不再占用产品界面。
- 左侧底部显示轻量版本号：只显示 `vX.Y.Z`，方便日常截图反馈问题。
- 工作台首页不放大号版本信息，避免干扰任务和收入这些核心数据。
