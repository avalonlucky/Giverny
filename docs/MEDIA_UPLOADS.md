# 大文件上传与媒体压缩方案

## 当前已上线能力

Giverny 的任务过程附件和验收附件统一使用以下上传策略：

- 单文件上限为 **200MB**。
- 12MB 以内文件使用普通上传。
- 12MB 以上文件自动使用 R2 Multipart Upload，按 8MB 分片上传。
- 前端显示总体上传进度；取消或失败时会中止分片任务，避免遗留未完成上传。
- 前端选取文件时校验大小，Worker 在普通上传、分片初始化和分片完成三个入口再次校验。
- 支持图片、PDF、Office、设计源文件、压缩包，以及 MP4、MOV、WebM、M4V、OGV 视频。

分片上传使每个 Worker 请求只携带一个小分片，200MB 文件不会触发 Cloudflare Worker 单次请求体上限。文件最终直接保存在 R2，D1 只记录文件元数据和任务关联。

## 为什么不能直接压缩任意文件

“500MB 压到 200MB”不是通用 ZIP 压缩问题。MP4、MOV、PDF、JPG 等格式本身已经压缩，再打 ZIP 通常只能减少很少体积。可靠的视频压缩需要重新编码画面和声音，并根据时长、分辨率、帧率和目标体积计算码率。

浏览器内运行 FFmpeg 处理 500MB 视频会大量占用内存和 CPU，容易造成页面无响应，也无法保证移动端稳定，因此不应放在网站前端执行。

## 推荐的媒体压缩架构

后续使用独立的 **Cloudflare Container + FFmpeg** 提供异步媒体处理服务：

1. 前端将 200MB–500MB 原文件按分片上传到 R2 临时区。
2. Worker 创建压缩任务，记录 `queued / processing / completed / failed / cancelled` 状态。
3. Container 从 R2 流式读取原文件，先用 `ffprobe` 获取时长、分辨率、帧率和音轨信息。
4. 以约 190MiB 为安全目标计算总码率，预留 MP4 容器、音频和元数据开销。
5. FFmpeg 使用 H.264 + AAC 输出 MP4，并启用 `faststart`；必要时执行两遍编码，提高目标体积的可控性。
6. 压缩结果写回 R2，Worker 校验实际大小、时长和可播放性。
7. 前端展示压缩前后体积、预计清晰度变化，并由用户确认采用压缩版。
8. 压缩成功前保留原文件；成功并确认后再按保留策略清理临时原件。

目标视频码率可按以下方式估算：

```text
目标总码率 = (目标字节数 × 8 ÷ 视频时长秒数)
目标视频码率 = 目标总码率 - 音频码率 - 容器安全余量
```

系统需要设置最低可接受视频码率。若 200MB 无法在可接受清晰度下容纳完整视频，应明确提示用户提高目标体积或降低分辨率，不能静默输出严重失真的文件。

## 实施前提

- Cloudflare 账户启用 Workers Paid 和 Containers。
- 本地 Docker / Colima 可用，用于构建并首次部署 Container 镜像。
- 新增媒体压缩任务表、临时 R2 对象生命周期和失败重试机制。
- 为视频处理设置 CPU、内存、临时磁盘、执行时长和并发上限。
- 补齐真实大视频回归：500MB 上传、断点失败、压缩成功、超时重试、取消、清理、成品播放与下载。

在这些条件完成前，产品只承诺 200MB 以内的可靠分片上传，不展示不可用的“自动压缩”按钮。

## 不采用的方案

- **ZIP / 7z**：对视频、PDF、JPG 等已压缩格式收益很低。
- **浏览器 FFmpeg**：500MB 文件的内存、耗时和兼容性不可控。
- **Cloudflare Stream 作为验收文件库**：适合在线播放和转码，但不适合作为需要保留、下载原始交付件的统一附件存储。
