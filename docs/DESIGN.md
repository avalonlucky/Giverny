# UI 视觉排版规范

> 本规范聚焦「视觉层级与信息密度」，与 [UX_OPTIMIZATION_AUDIT.md](./UX_OPTIMIZATION_AUDIT.md)（讲交互流程）互补。
> 任何 AI 或人在修改界面布局、新增列表/卡片前，都应先读这份文档。

更新时间：2026-06-21

## 零、视觉基准：参考稿的纸张体系

当前视觉基准是用户提供的 `/Users/luban/Desktop/giverny-redesign/index.html`。修改界面时必须先读对应源码，再接入真实业务；不能只凭截图做近似版。

1. **业务容器使用直角纸张**：页面区块、详情面板、弹窗、卡片、菜单和输入表面默认直角，不使用装饰性圆角。
2. **全站不使用投影建立层级**：禁止卡片阴影、弹窗阴影和悬停浮起。层级只靠纸张底色、遮罩、细发丝线和留白建立。
3. **标题保持轻量**：页面与弹窗标题通常使用衬线字体、`font-weight: 600`；普通分区标题使用常规或中等字重，禁止动辄 800/900。
4. **按钮保持轻量**：主按钮只承担一个明确主操作；次按钮优先使用无边框文字按钮。普通按钮不做厚重描边、阴影或大圆角。
5. **胶囊只保留语义用途**：设计类型、状态标签、进度档位和开关可以继续使用胶囊形，因为形状本身承担选中/状态语义；不能把整个面板、普通按钮或输入框做成胶囊。
6. **纸张层级**：页面底使用 `--color-surface-alt`，主内容使用 `--color-surface-strong` / `--color-editorial-surface`，可编辑输入区使用略深的 `--color-surface-alt`；边界使用 `--color-border` 发丝线。

## 一、核心原则：唯一焦点

每一行 / 每一块卡片**只保留一个视觉焦点**。其余信息按「主 → 次 → 状态」三层逐级降权。

**自检方法**：眯起眼睛看，第一眼落到的元素，必须是这张卡片的主体——
- 任务列表里 → 任务名
- 统计卡片里 → 数值
- 详情面板里 → 标题

如果第一眼看到的是日期、类型、辅助标签，说明层级错了。

## 二、五层信息分级

| 层级 | 内容 | 视觉处理 | 用哪个 token |
|---|---|---|---|
| **主信息** | 任务名、标题、数值 | 最深色 + 粗体 800 + 较大字号 | `var(--color-text)` |
| **次信息** | 日期、类型、对接人、需求描述、工时 | 灰色 + 常规字重 + 12px | `var(--color-text-secondary)` 或 `--color-text-muted` |
| **状态信号** | 临期、验收状态、进度 | 强调色 + badge / 进度条，**集中归组** | `var(--color-primary)` / `--color-primary-strong` |
| **管理员专属信息** | 普通成员、甲方预览、公开只读链接不可见的内部细节 | 棕色 + 不抢主焦点，统一使用 `admin-only-data` | `var(--color-admin-only)` |
| **公开解释标记** | 补录等需要对甲方解释来源的公开提示 | 非棕色公开标签，不使用 `admin-only-data` | `var(--color-supplement)` |

## 三、硬规则

1. **任务名是唯一焦点** —— 不要让日期、类型等辅助信息用同等的粗细或深色。
2. **辅助信息一律降权** —— 灰字、小号；放不下时下沉到副行，而不是抢主行的视觉权重。
3. **状态信号收敛成一组** —— 临期 / 验收 / 进度归到右侧一块，用虚线分隔或留白和正文区分开，不要散落三处。
4. **日期不抢首位** —— 在列表型布局里，日期是检索辅助、不是主语；用灰字，必要时挪到副行。
5. **跨视图呈现要一致** —— 同一类信息（如临期）在工作台、任务管理、日历等所有视图里用**同一种样式**（都做 badge 就都 badge），不要一个 badge 一个彩色文字。
6. **窄列不塞多元素** —— 单个 grid 列里只放一种信息；临期 badge 之类的东西不要挤进日期列，独立成位。
7. **内部可见信息统一棕色** —— 任何只给管理员看的字段，都用 `admin-only-data` / `--color-admin-only`；普通成员、甲方预览、公开只读报告不显示该字段。
8. **补录是公开解释，不是内部信息** —— 「补录」必须让甲方可见，用 `--color-supplement`，不要使用棕色管理员专属样式。
9. **禁止浏览器原生弹窗** —— 不使用 `window.alert` / `window.confirm` / `window.prompt`；所有确认、提示、输入都走站内 modal / toast / form，避免系统弹窗破坏产品一致性。
10. **新增组件先查复用** —— 新增任何按钮组、弹窗、选择器、滑杆、上传区、列表行、状态 badge 前，必须先在 `src/App.tsx` / `src/App.css` 搜索现有实现；同类能力优先复用或抽成共享组件，禁止写一套风格不一致的临时组件。
11. **没有现成组件也不能脱离风格** —— 确实没有可复用组件时，新组件也必须沿用当前产品的 token、纸张表面、发丝线、留白、轻标题、轻按钮和交互反馈规则；不允许出现浏览器默认样式或明显不属于 Giverny 的“自创丑组件”。
12. **输入区与附件区使用统一表面** —— 弹窗内的多行输入、文件上传和同类可编辑区域，默认使用比弹窗背景略深的 `var(--color-surface-alt)` 作为直角底色，并使用相同的内边距层级；不要在同一表单内混用纯白输入框、灰底附件框和另一套边框语言。
13. **日期选择统一复用 `PlanDateTimeField`** —— 默认使用同一套「月份标题 + 月份切换 + 日历 + 小时/分钟」交互；纯日期场景传 `includeTime={false}` 隐藏右侧时间列。弹层优先在所属弹窗内完整显示；空间不足时允许浮层越界，不能为了裁切而牺牲日期、时间或“今天 / 清除”等操作。

## 四、复用现有的 CSS token（不要新造）

本项目 `App.css` 顶部 `:root` 已定义完整色板，**新增样式必须复用，禁止裸色值**：

```css
/* 文字三档 —— 对应「主 / 次 / 弱」三层信息 */
--color-text:           #17282f;   /* 主信息 */
--color-text-secondary: #5f6d70;   /* 次信息 */
--color-text-muted:     #66716d;   /* 弱信息 / 占位，小字号仍保持可读 */

/* 品牌与状态 */
--color-primary:        #2f6f6d;   /* 进度条 / 强调 */
--color-primary-strong: #1f4751;
--color-admin-only:     #8a5f2a;   /* 仅管理员可见的内部信息 */
--color-supplement:     #2f6f6d;   /* 补录等对外可见的解释标记 */

/* 面板与表面 */
--color-surface:        #f8f9f5;
--color-surface-strong: #fdfdfb;
--color-surface-alt:    #f2f3ee;
--color-sidebar:        #eeefe9;
```

> 状态色（临期黄、逾期红）目前在 `.due-tag` / `.status-*` 等组件内联，未来如需统一，
> 可新增 `--color-warn-soon` / `--color-warn-over`，但**不要破坏上面这套文字三档**。

## 五、组件命名约定（复用 = 继承规则）

| 类名模式 | 含义 | 视觉默认 |
|---|---|---|
| `*-row` | 列表行（任务行 / 管理行） | `--color-surface-alt` 直角纸张底 |
| `*-main` | 主信息区（**唯一焦点**，深色粗体放这） | `--color-text` + font-weight 800 |
| `*-meta` / `*-muted` | 次信息区（灰字小号） | `--color-text-secondary` + 12px |
| `*-state` / `status-*` | 状态归组区（右侧一组） | badge / 进度条，虚线分隔 |
| `due-tag` | 临期/逾期标签（**跨视图统一复用**） | 语义胶囊 badge + 状态色底 |

## 六、复用优先级

新增 UI 前按这个顺序处理：

1. **先搜现有组件 / 类名**：用 `rg "关键词|className|组件名" src/App.tsx src/App.css` 查找类似功能，例如进度先搜 `progress-slider-row` / `progress-quick-options`，确认先搜 `ConfirmDialog`，弹窗先搜 `ModalShell`。
2. **能复用就复用**：同类功能只允许有一套主要视觉语言；例如进度编辑、档位快选、验收确认、文件上传、状态 badge 不应在不同弹窗里长得不一样。
3. **需要变化就扩展**：如果场景略有不同，优先给现有组件加 modifier class 或抽出小组件，而不是复制一段再改成另一套样式。
4. **确实没有再新建**：新建时必须复用 `--color-*` token、现有按钮 / 输入框 / badge / modal / toast 的尺寸和交互反馈，不使用浏览器默认控件外观作为最终方案。

## 七、改 UI 前的自检清单

- [ ] 这一行/这块的焦点是谁？只有它用了最深色 + 粗体吗？
- [ ] 辅助信息（日期/类型/对接人）是不是都灰字小号了？
- [ ] 状态信号有没有散落？能不能归到右侧一组？
- [ ] 同类信息在不同视图的呈现是否一致？
- [ ] 新增组件前是否搜索过现有实现？是否能复用或抽成共享组件？
- [ ] 如果没有可复用组件，新组件是否沿用当前 token、按钮、输入框、弹窗、滑杆、badge 等产品语言？
- [ ] 面板、弹窗、菜单和普通按钮是否保持直角、无阴影？圆角是否只用于状态 / 分类 / 开关等语义胶囊？
- [ ] 标题和按钮是否足够轻，避免无意义的 800/900 字重？
- [ ] 普通成员、甲方预览、公开只读链接不可见的内部信息，是否用了 `admin-only-data` 棕色标记？
- [ ] 补录等需要对甲方解释来源的公开标签，是否避免使用管理员专属棕色？
- [ ] 删除、确认、输入等交互是否都用了站内组件，而不是浏览器原生弹窗？
- [ ] 有没有窄列里塞了多个元素导致挤压 / ellipsis？
- [ ] 新写的色值是不是都用了 `var(--color-*)`，而不是裸 hex？
- [ ] 新增日期输入是否复用了 `PlanDateTimeField`，并按需决定是否显示时间列？
