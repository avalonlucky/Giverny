## 이것은 무엇인가

Giverny는 또 하나의 범용 프로젝트 관리 도구가 아닙니다. “협업 작업 + 공수 증거 + 납품 검수 + 월간 정산”을 다루는 가벼운 운영 백오피스이며, 디자인, 콘텐츠, 운영, 외주, 내부 협업 작업을 요구부터 정산까지 완전하게 남기고 싶은 팀에 적합합니다.

단순히 작업이 어떤 보드 열로 이동했는지가 아니라, 각 작업이 최종적으로 다음 질문에 답할 수 있는지를 중요하게 봅니다.

- 이 일은 누가 요청했고, 누가 담당하며, 언제 납품해야 하나요?
- 과정에서 어떤 진척이 있었고, 어떤 파일이 업로드되었나요?
- 실제로 몇 시간이 투입되었고, 어떤 공수가 정산 가능한가요?
- 최종적으로 누가 검수했고, 검수 근거는 무엇인가요?
- 월말에 고객이나 회사에 작업, 공수, 파일, 금액을 바로 보여줄 수 있나요?

## 핵심 기능

| 기능 | 해결하는 문제 |
| --- | --- |
| 작업 흐름 | 계획 중, 진행 중, 검수 대기, 검수 완료까지 상태를 기록 |
| 과정 협업 | 작업 오른쪽에서 진행, 첨부, 시간 구간, 동적 타임라인을 기록 |
| 공수 관리 | 모든 통계, 수입, 정산은 실제 공수만 기준으로 사용 |
| 납품 검수 | 최종 검수 패널에서 기본 정보, 진행률, 분할 공수, 첨부, 비고 확인 |
| 월간 정산 | 정산 월별로 공수, 수입, 검수 상황, 연간 추세를 집계 |
| 고객 대사 | 읽기 전용 링크로 월간 보고서, 작업 상세, 납품 파일을 공유 |
| 파일 보관 | 과정 파일, 검수 첨부, 최종본을 작업 생명주기에 남김 |

## 적합한 시나리오

- 회사 내부 디자인, 콘텐츠, 운영, 마케팅 작업에 여러 사람의 협업과 월간 리뷰가 필요한 경우.
- 프리랜서 디자이너나 외주 팀이 실제 공수, 납품 파일, 정산 근거를 기록해야 하는 경우.
- 책임자가 월별로 “무엇을 했고, 얼마나 걸렸고, 무엇을 납품했고, 얼마를 정산해야 하는지” 확인해야 하는 경우.
- 클라이언트나 협업자가 채팅 기록 대신 읽기 전용 페이지에서 월간 보고서와 납품 내용을 확인해야 하는 경우.

## 프로젝트 포지션

Giverny는 디자인 프리랜스 업무를 관리하는 운영 백오피스입니다. 작업 요구사항, 디자인 과정, 실제 공수, 과정 파일, 검수 첨부, 월간 정산, 클라이언트 읽기 전용 대사 링크를 하나의 시스템에 넣어 채팅, 스프레드시트, 클라우드 드라이브, 월말 수작업 정산 사이를 반복해서 오가는 일을 줄입니다.

현재 프로덕션 사이트 [mayeai.com](https://mayeai.com)에서 운영됩니다. 실제 운영 데이터는 Cloudflare D1에 저장하고, 업로드 파일은 Cloudflare R2에 저장합니다. 스테이징 사이트는 중단되었고, 앞으로는 로컬 검증 후 바로 프로덕션을 업데이트합니다.

Giverny는 범용 프로젝트 관리 도구가 아닙니다. 디자인 작업의 생명주기를 중심으로 과정, 공수, 파일, 검수, 월간 보고서를 연결하는 “디자인 서비스 정산 워크벤치”에 가깝습니다.

## 다국어 버전

README 상단에는 GitHub 기본 방식의 언어 진입점이 있습니다. 简体中文, 繁體中文, English, 日本語, 한국어는 각각 독립 Markdown README 파일로 열리며 외부 모음 페이지로 이동하지 않습니다.

각 언어 버전은 Giverny가 무엇인지, 왜 Notion / Feishu / Excel만으로 부족한지, 핵심 워크플로우, 주요 업무 규칙, 기술 아키텍처, 릴리스 규칙을 같은 구조로 설명해야 합니다.

## 왜 Giverny가 필요한가

디자인 프리랜스 업무의 흔한 문제는 “작업 관리 도구가 없다”가 아니라 증거와 정보가 너무 많은 곳에 흩어져 있다는 점입니다.

- 요구사항은 WeChat, Feishu, 이메일, 구두 커뮤니케이션에 흩어져 있습니다.
- 공수는 임시 Excel이나 채팅 메모에 남습니다.
- 과정 파일과 최종 파일이 클라우드 드라이브나 로컬 폴더에 섞입니다.
- 월말에 클라이언트가 궁금한 것은 무엇을 했는지, 얼마나 걸렸는지, 무엇을 납품했는지, 왜 이 금액을 정산해야 하는지입니다.
- 디자이너도 언제 요청을 받았고, 언제 수정했고, 어떤 파일이 최종 검수 근거인지 기록해야 합니다.

Giverny는 이 흩어진 정보를 추적 가능한 하나의 흐름으로 만듭니다.

```text
작업 요구사항 -> 과정 진행 -> 시간 기록 -> 파일 보관 -> 납품 검수 -> 월간 정산 -> 클라이언트 읽기 전용 대사
```

## 누구를 위한 것인가

| 역할 | 주요 요구 | Giverny가 제공하는 것 |
| --- | --- | --- |
| 프리랜서 디자이너 | 각 요구, 수정 과정, 실제 공수, 검수 파일을 명확히 기록 | 작업 상세, 진행 타임라인, 분할 공수, 검수 첨부, 월간 통계 |
| 디자인 서비스 담당자 | 월별 작업량, 수입, 파일, 정산 상태를 확인 | 대시보드, 수입 통계, 파일 라이브러리, 월간 보고서, PDF |
| 클라이언트 / 협업자 | 최종 월간 보고서, 작업 상세, 납품 파일만 확인 | `/share/:token` 읽기 전용 링크 |
| 인수 개발자 / AI 코딩 어시스턴트 | 코드 구조, 업무 규칙, 릴리스 규칙을 빠르게 이해 | `AGENTS.md`, `docs/`, `handoff/`, Releases |

## 대표 사용 시나리오

### 시나리오 1: 이번 달 일반 작업

1. 현재 월에 작업을 만들고 작업명, 디자인 유형, 요구사항, 예정 시작, 예정 납품을 입력합니다.
2. 작업을 수락한 뒤 오른쪽 “진행” 패널에 “사이즈 확인 완료, 초안 작업 중” 같은 기록을 남깁니다.
3. 실제 투입을 기록해야 하면 `09:00 - 10:30 초안` 같은 시간 구간을 추가합니다.
4. 과정 첨부나 최종 출력 파일을 업로드합니다.
5. 납품 시 “납품 검수”를 펼쳐 기본 정보, 진행률, 공수, 첨부, 비고를 확인합니다.
6. 검수 후 작업은 이번 달 공수, 수입, 월간 보고서에 포함됩니다.

### 시나리오 2: 과거 작업 보충 기록

1. 작업 생성 시 “보충 기록”을 켭니다.
2. 작업 날짜는 실제 발생일, 예를 들어 5월 20일로 입력합니다.
3. 정산 월은 포함되어야 하는 월, 예를 들어 2026년 6월을 선택합니다.
4. 클라이언트는 공개 “보충” 태그를 보고 왜 이 작업이 이번 달 정산에 들어왔는지 이해할 수 있습니다.
5. 보충은 공개 설명 정보이며 관리자 전용 갈색 정보로 만들면 안 됩니다.

### 시나리오 3: 작업은 끝났지만 아직 검수 전

1. 상태는 “검수 대기”로 유지합니다.
2. 작업 오른쪽에서 실제 공수, 납품 첨부, 검수 비고를 보완합니다.
3. “검수하러 가기”를 눌러 최종 검수 팝업을 엽니다.
4. 확인 후 상태는 “검수 완료”, 진행률은 자동으로 100%, 공수는 잠겨 정산에 들어갑니다.

### 시나리오 4: 클라이언트 월말 대사

1. 관리자가 월간 보고서 / 정산에서 월 데이터를 잠급니다.
2. 읽기 전용 공유 링크를 생성합니다.
3. 클라이언트는 링크에서 이번 달 작업, 공수, 납품 파일, 정산 금액을 확인합니다.
4. 클라이언트 페이지에는 편집, 삭제, 검수 같은 관리 기능이 표시되지 않습니다.

## 핵심 워크플로우

![Giverny task lifecycle](./docs/assets/readme/workflow.svg)

1. **작업 생성**: 작업명, 디자인 유형, 요구사항, 예정 시작, 예정 납품, 담당자, 정산 월을 기록합니다.
2. **과정 진행**: 오른쪽 “진행” 패널에서 진행 기록, 과정 첨부, 시간 구간, 상태, 전체 진행률을 관리합니다.
3. **공수 축적**: 모든 통계는 실제 공수를 기준으로 합니다. 예정 시작과 예정 납품은 일정 참고용이며 분석, 공수 계산, 정산에 참여하지 않습니다.
4. **납품 검수**: 기본 정보, 진행률, 분할 공수, 검수 첨부, 비고를 확인합니다. 확인 후 검수 완료, 진행률 100%, 공수가 정산에 들어갑니다.
5. **월간 정산**: 정산 월별로 공수, 수입, 검수 상태, 연간 추세를 집계하고 클라이언트 읽기 전용 링크와 PDF를 생성합니다.

## 모듈 상세

### 작업대

작업대는 월별 운영 요약입니다. 핵심 질문은 “이번 달 얼마나 했고, 얼마를 정산할 수 있으며, 아직 검수되지 않은 것은 무엇인가?”입니다. 월 총공수, 청구 공수, 예상 수입, 검수 상황, 지연 알림, 작업 명세, 월간 인사이트, 연간 통계를 표시합니다.

핵심 기준: 작업대의 모든 통계는 `settlement_month`와 실제 공수를 기반으로 하며, 예정 시작 / 예정 납품은 정산에 사용하지 않습니다.

### 작업 네비게이션

작업 네비게이션은 일상 작업 유지보수 화면입니다. 왼쪽은 목록 또는 달력이고 오른쪽은 선택된 작업 상세입니다. 목록은 상태 필터, 작업명, 요구 요약, 담당자, 납품 시간, 공수, 상태를 표시합니다. 보충 작업은 공개 “보충” 태그를 표시합니다.

정보 탭은 작업명, 디자인 유형, 예정 시작, 예정 납품, 정산 월, 요청자, 담당자, 검수자, 요구사항을 보여줍니다. 관리자 전용 내부 정보는 갈색 `admin-only-data`를 사용합니다.

진행 탭은 진행 기록, 과정 첨부, 시간 기록, 10% 단계의 전체 진행률, 확인 후 저장 방식, 동적 타임라인, 납품 검수를 포함합니다.

### 납품 검수

검수는 “진행 중”에서 “정산”으로 넘어가는 경계입니다. 최종 검수 팝업은 기본 정보, 현재 진행률, 분할 공수, 검수 파일, 선택 비고를 확인하게 합니다.

검수 후 작업은 검수 완료가 되고, 실제 공수는 정산에 잠기며, 진행률은 100%가 되고, 작업대 / 수입 / 연간 통계 / 월간 보고서에 반영됩니다.

### 파일 라이브러리

파일 라이브러리는 별도 클라우드 드라이브가 아니라 작업 생명주기의 파일 보관 화면입니다. 과정 파일, 진행 첨부, 검수 첨부, 최종본을 작업 / 프로젝트 단위로 정리합니다.

잘못 업로드했거나 중복된 파일은 삭제할 수 있지만 반드시 앱 내 확인을 사용하고, D1 첨부 기록, R2 원본 파일과 미리보기, 감사 로그를 함께 처리해야 합니다.

### 수입과 월간 보고서

수입 페이지는 수입 추세와 세후 추정을 확인합니다. 월간 보고서는 잠긴 월 데이터를 읽어 읽기 전용 링크를 생성합니다. 클라이언트는 월간 작업 요약, 작업 상세, 공수, 정산 금액, 납품 파일, PDF 내용을 볼 수 있습니다.

### 설정

설정에서는 접근 암호, 디자인 유형, 시급과 세금 계산 방식, PDF 헤더, 서비스 회사명, 계정 보안, 시스템 정보, Cloudflare 바인딩, 백업 진입점을 관리합니다.

## 데이터 기준과 정산 규칙

### 월 귀속

작업의 월 귀속은 `settlement_month`만 봅니다. 일반 작업은 현재 선택 월에 속하고, 보충 작업은 실제 발생일과 정산 월을 분리할 수 있습니다. 작업대, 작업 목록, 수입 통계, 월간 보고서는 모두 `settlement_month`로 집계합니다.

### 실제 공수

실제 공수는 분할 시간 기록, 검수 시 확인한 실제 공수, 검수 완료 후 잠긴 공수에서 옵니다. 월간 인사이트, 수입 추정, 월간 보고서, 연간 통계, 정산 PDF에 사용됩니다.

### 예정 시작 / 예정 납품

예정 시작과 예정 납품은 일정과 알림용입니다. 임박 / 지연 판단, 정렬, 달력 보기에는 사용할 수 있지만 공수 통계, 수입 계산, 정산 월 보완에는 사용하지 않습니다.

## 파일 라이프사이클

```text
과정 파일 업로드 -> 파일 라이브러리 진입 -> 용도 / 태그 표시 -> 검수 첨부 또는 최종본 -> 월간 읽기 전용 공유 -> 장기 보관 또는 수동 정리
```

현재 자동 정기 정리 기능은 없습니다. 과거 첨부가 월간 보고서, 클라이언트 공유 페이지, 회고, 재수정에서 계속 참조될 수 있기 때문입니다. 프로덕션 운영에서는 직접 삭제보다 보관을 우선합니다.

## FAQ

### 왜 Notion / Feishu 표 / Excel만 쓰지 않나요?

정보를 저장할 수는 있지만 진행, 실제 공수, 첨부, 검수, 정산, 클라이언트 읽기 전용 링크를 감사 가능한 하나의 체인으로 안정적으로 연결하기 어렵기 때문입니다.

### 왜 작업을 직접 삭제하면 안 되나요?

작업은 공수, 수입, 월간 보고서, 과거 대사에 영향을 줍니다. 직접 삭제하면 이력이 깨질 수 있으므로 비정상 작업은 보류, 종료, 무효 상태로 설명을 남겨야 합니다.

### 왜 예정 납품은 정산에 참여하지 않나요?

예정 시간은 계획이지 실제 투입이 아닙니다. 정산은 실제 공수와 검수 상태를 기준으로 해야 합니다.

### 왜 보충 기록을 클라이언트에게 보여줘야 하나요?

보충 기록은 “왜 이 작업이 이번 달 정산에 들어왔는지” 설명합니다. 숨기면 클라이언트에게 갑자기 생긴 작업처럼 보일 수 있습니다.

### 왜 관리자 전용 정보는 갈색인가요?

갈색은 내부 정보 표시입니다. 관리자는 갈색을 보면 그 정보가 일반 구성원, 클라이언트 미리보기, 공개 읽기 링크에 나오지 않는다는 것을 알 수 있습니다.
