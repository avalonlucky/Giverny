## これは何か

Giverny は、もう一つの汎用プロジェクト管理ツールではありません。「協働タスク + 工数証跡 + 納品検収 + 月次決済」を扱う軽量な運用バックオフィスであり、デザイン、コンテンツ、運用、外注、社内協働タスクを要件から決済まで残したいチームに向いています。

単にタスクが看板の別列へ移動したかを見るのではなく、各仕事について次の問いに答えられるかを重視します。

- 誰が依頼し、誰が担当し、いつ納品するのか？
- プロセスで何が進み、どのファイルがアップロードされたのか？
- 実際に何時間かかり、どの工数が決済対象なのか？
- 最終的に誰が検収し、検収根拠は何か？
- 月末にクライアントや会社へ、タスク、工数、ファイル、金額をすぐ提示できるか？

## コア機能

| 機能 | 解決する問題 |
| --- | --- |
| タスクフロー | 計画中、進行中、検収待ち、検収済みまで状態を記録 |
| プロセス協働 | タスク右側で進捗、添付、時間帯、動的タイムラインを記録 |
| 工数管理 | 分析、収入、決済は実工数だけを基準にする |
| 納品検収 | 最終確認パネルで基本情報、進捗、分割工数、添付、備考を確認 |
| 月次決済 | 決済月ごとに工数、収入、検収状況、年間傾向を集計 |
| クライアント照合 | 読み取り専用リンクで月報、タスク明細、納品ファイルを共有 |
| ファイル保管 | 作業ファイル、検収添付、最終稿をタスクライフサイクルに残す |

## 向いているシーン

- 社内のデザイン、コンテンツ、運用、マーケティングタスクで、複数人の協働と月次レビューが必要な場合。
- フリーランスデザイナーや外注チームが、実工数、納品ファイル、決済根拠を残したい場合。
- 責任者が「何をしたか、何時間かかったか、何を納品したか、いくら決済すべきか」を月ごとに確認したい場合。
- クライアントや協力者が、チャット履歴ではなく読み取り専用ページで月報と納品内容を確認したい場合。

## プロジェクトの位置づけ

Giverny は、デザインの副業・業務委託を管理するための運用バックオフィスです。タスク要件、制作プロセス、実工数、作業中ファイル、検収添付、月次決済、クライアント向け読み取り専用照合リンクを一つのシステムに集約します。

現在は本番サイト [mayeai.com](https://mayeai.com) で運用されています。業務データは Cloudflare D1、アップロードファイルは Cloudflare R2 に保存します。ステージングサイトは廃止済みで、今後はローカル検証後に本番へ直接更新します。

Giverny は汎用プロジェクト管理ツールではありません。デザインタスクのライフサイクルを中心に、進捗、工数、ファイル、検収、月報をつなぐ「デザインサービス決済ワークベンチ」です。

## なぜ Giverny が必要か

デザイン副業でよく起きる問題は「タスク管理ツールがない」ことではなく、証跡が多くの場所に分散していることです。

- 要件は WeChat、Feishu、メール、口頭確認に散らばる。
- 工数は一時的な Excel やチャットのメモに残る。
- 作業ファイルと最終稿がクラウドドライブやローカルフォルダに混ざる。
- 月末にクライアントが知りたいのは、何をしたか、何時間かかったか、何を納品したか、なぜその金額を支払うのか。
- デザイナー自身も、いつ依頼を受け、いつ変更し、どのファイルが最終検収根拠なのかを残したい。

Giverny はこれらを追跡可能な一本の流れにします。

```text
タスク要件 -> プロセス進捗 -> 時間記録 -> ファイル保管 -> 納品検収 -> 月次決済 -> クライアント読み取り専用照合
```

## 対象ユーザー

| ロール | 主なニーズ | Giverny が提供するもの |
| --- | --- | --- |
| フリーランスデザイナー | 各要件、修正、実工数、検収ファイルを明確に残す | タスク詳細、進捗タイムライン、分割工数、検収添付、月次統計 |
| デザインサービス責任者 | 月ごとに作業量、収入、ファイル、決済状態を確認する | ダッシュボード、収入統計、ファイル庫、月報、PDF |
| クライアント / 協力者 | 最終月報、タスク明細、納品ファイルだけを確認する | `/share/:token` 読み取り専用リンク |
| 引き継ぎ開発者 / AI コーディング助手 | コード構造、業務ルール、リリース規律を早く理解する | `AGENTS.md`、`docs/`、`handoff/`、Releases |

## 代表的な利用シーン

### シーン 1：当月の通常タスク

1. 当月にタスクを作成し、タスク名、デザイン種別、要件、予定開始、予定納品を入力する。
2. タスクを受けたら右側の「進捗」パネルに「サイズ確認済み、ラフ作成中」などを記録する。
3. 実作業を記録する必要がある場合、`09:00 - 10:30 初稿` のように時間帯を追加する。
4. 作業中の添付や最終出力ファイルをアップロードする。
5. 納品時に「納品検収」を展開し、基本情報、進捗、工数、添付、備考を確認する。
6. 検収後、そのタスクは当月の工数、収入、月報に入る。

### シーン 2：過去タスクの補録

1. タスク作成時に「補録」をオンにする。
2. タスク日付は実際に発生した日、たとえば 5 月 20 日を入れる。
3. 決済月は計上したい月、たとえば 2026 年 6 月を選ぶ。
4. クライアントは公開の「補録」タグを見て、なぜ当月決済に入っているか理解できる。
5. 補録は公開説明情報であり、管理者専用の茶色表示にしてはいけない。

### シーン 3：作業済みだが未検収

1. ステータスは「検収待ち」にしておく。
2. 右側で実工数、納品添付、検収備考を補う。
3. 「検収へ」をクリックして最終確認ダイアログを開く。
4. 確認後、ステータスは「検収済み」、進捗は自動で 100%、工数はロックされ決済に入る。

### シーン 4：月末のクライアント照合

1. 管理者が月報 / 決済で月次データをロックする。
2. 読み取り専用リンクを生成する。
3. クライアントはリンクから当月タスク、工数、納品ファイル、決済金額を確認する。
4. クライアントページには編集、削除、検収などの管理操作は表示しない。

## コアワークフロー

![Giverny task lifecycle](./docs/assets/readme/workflow.svg)

1. **タスク作成**：タスク名、デザイン種別、要件、予定開始、予定納品、担当者、決済月を記録する。
2. **プロセス進行**：右側の「進捗」パネルで進捗、添付、時間帯、ステータス、全体進捗を管理する。
3. **工数の蓄積**：すべての統計は実工数が基準。予定開始と予定納品はスケジュール参考のみで、分析、工数計算、決済には使わない。
4. **納品検収**：基本情報、進捗、分割工数、検収添付、備考を確認する。確認後は検収済み、進捗 100%、工数が決済へ入る。
5. **月次決済**：決済月ごとに工数、収入、検収状況、年間傾向を集計し、クライアント向け読み取り専用リンクと PDF を生成する。

## モジュール詳細

### ダッシュボード

ダッシュボードは月ごとの運用総覧です。「今月どれだけ作業したか、いくら決済できるか、何が未検収か」を確認します。本月総工数、請求対象工数、見込み収入、検収状況、期限超過提醒、タスク明細、月次インサイト、年間統計を表示します。

重要な基準：ダッシュボードの統計は `settlement_month` と実工数に基づき、予定開始 / 予定納品は決済に使いません。

### タスクナビゲーション

タスクナビゲーションは日常のタスク保守画面です。左側はリストまたはカレンダー、右側は選択中タスクの詳細です。リストでは状態フィルタ、タスク名、要件概要、担当者、納品時刻、工数、状態を表示します。補録タスクには公開の「補録」タグを表示します。

情報タブはタスク名、デザイン種別、予定開始、予定納品、決済月、依頼者、担当者、検収者、要件を表示します。管理者専用の内部情報は茶色の `admin-only-data` を使います。

進捗タブは進捗記録、過程添付、時間記録、10% 刻みの全体進捗、確認してから保存する仕組み、動的タイムライン、納品検収を含みます。

### 納品検収

検収は「作業中」から「決済」へ移る境界です。最終確認ダイアログでは、基本情報、現在の進捗、分割工数、検収ファイル、任意の備考を確認します。

確認後、タスクは検収済みとなり、実工数は決済にロックされ、進捗は 100% になり、月報や収入統計に入ります。

### ファイル庫

ファイル庫は独立したクラウドドライブではなく、タスクライフサイクルのファイル保管ビューです。作業中ファイル、進捗添付、検収添付、最終稿をタスク / プロジェクト単位で整理します。

誤アップロードや重複ファイルは削除できますが、必ずアプリ内確認を使い、D1 の添付記録、R2 の原始ファイルとプレビューを整理し、監査ログに残します。

### 収入と月報

収入ページでは収入傾向と税引後見積もりを確認します。月報はロックされた月次データを読み取り、読み取り専用リンクを生成します。クライアントは月次タスク、明細、工数、決済金額、納品ファイル、PDF 内容を確認できます。

### 設定

設定では、アクセス口令、デザイン種別、時給と税計算方式、PDF ヘッダー、サービス会社名、アカウント安全、システム情報、Cloudflare バインディング、バックアップ入口を管理します。

## ロールと権限

| 機能 | 管理者 | 口令ユーザー | クライアント共有リンク |
| --- | --- | --- | --- |
| ダッシュボード閲覧 | はい | はい | いいえ |
| タスク作成 / 編集 | はい | 口令権限により制限 | いいえ |
| タスク動態削除 | はい | いいえ | いいえ |
| ファイルアップロード / 削除 | はい | 制限あり | いいえ |
| 検収確認 | はい | いいえ | いいえ |
| 口令生成 | はい | いいえ | いいえ |
| 管理者専用時間情報の閲覧 | はい | いいえ | いいえ |
| 月報と納品ファイル閲覧 | はい | はい | 読み取り専用 |

視覚ルール：管理者専用情報は茶色の `admin-only-data`、補録は公開説明タグ、クライアント読み取り専用ページには内部ボタン、削除入口、内部時間、管理状態を表示しません。

## データ基準と決済ルール

### 月の帰属

タスクの月帰属は `settlement_month` のみを見ます。通常タスクは現在選択月に属し、補録タスクは実際の日付と決済月を分けられます。ダッシュボード、タスクリスト、収入統計、月報はすべて `settlement_month` で集計します。

### 実工数

実工数は分割時間記録、検収時に確認した実工数、検収済みタスクのロック済み工数から来ます。月次インサイト、収入見積もり、月報、年間統計、決済 PDF に使います。

### 予定開始 / 予定納品

予定開始と予定納品はスケジュールと提醒のみです。期限間近 / 期限超過の判断、並び替え、カレンダー表示には使えますが、工数統計、収入計算、決済月の補完には使いません。

## ファイルライフサイクル

```text
作業ファイルをアップロード -> ファイル庫へ入る -> 用途 / タグを付ける -> 検収添付または最終稿 -> 月報読み取り専用共有 -> 長期保存または手動整理
```

現在は自動定期削除を行いません。過去の添付が月報、クライアント共有、振り返り、修正依頼で参照される可能性があるためです。本番運用では直接削除よりもアーカイブを優先します。

## デプロイとリリース

デプロイ前に [docs/DEPLOYMENT.md](./docs/DEPLOYMENT.md) を読んでください。現在 staging は維持しておらず、ローカル検証後に本番へ直接デプロイします。

```bash
env -u ALL_PROXY -u HTTPS_PROXY -u HTTP_PROXY -u all_proxy -u https_proxy -u http_proxy npm run deploy:worker
```

本番リソース：

- Worker: `designer-worklog`
- D1: `designer-worklog-db`
- R2: `designer-worklog-uploads`
- Domain: `mayeai.com` / `www.mayeai.com`

バージョン、tag、GitHub Release が必要なのは、サイト本体に影響する本番更新だけです。README、リポジトリ紹介、多言語説明、handoff、内部協作文書は通常の commit / push のみで十分です。

サイト本体に影響する更新では、バージョンファイル、`CHANGELOG.md`、lint / build、本番デプロイ、commit / push、tag、GitHub Release、必要な UI スクリーンショットをそろえます。

## FAQ

### なぜ Notion / Feishu 表 / Excel だけではだめですか？

情報は保存できますが、進捗、実工数、添付、検収、決済、クライアント読み取り専用リンクを監査可能な一本のチェーンに安定してつなぐのは難しいためです。

### なぜタスクを直接削除できないのですか？

タスクは工数、収入、月報、過去の照合に影響します。直接削除すると履歴が壊れるため、異常タスクは一時停止、終了、無効化で説明を残します。

### なぜ予定納品は決済に使わないのですか？

予定は計画であり、実際の投入ではありません。決済は実工数と検収状態に基づく必要があります。

### なぜ補録をクライアントに見せるのですか？

補録は「なぜこのタスクが当月決済に入っているか」を説明するためです。隠すと、クライアントには突然増えたタスクに見えます。

### なぜ管理者専用情報は茶色ですか？

茶色は内部情報の目印です。管理者は茶色を見れば、その情報が通常メンバー、クライアントプレビュー、公開読み取りリンクに出ないと分かります。
